## Codebase Patterns
- Run backend quality checks from `backend/` with `bun run typecheck`, `bun run lint`, and `bun test`.
- Keep shared Elysia setup in `backend/src/app.ts`, register domain routes through `backend/src/modules/index.ts`, and keep `backend/src/server.ts` limited to bootstrapping the process.
- Keep backend database wiring lazy: parse DB env in `backend/src/config/env.ts`, create the client in `backend/src/db/client.ts`, and run SQL files from `backend/src/db/migrations/` so automated tests do not need a live PostgreSQL instance.
- The SQL migration runner executes each migration with PostgreSQL `search_path` set to `DATABASE_SCHEMA`, so unqualified tables should be written for the configured app schema instead of `public`.
- Better Auth currently stores authenticated user IDs as `text` in the `"user"` table, so app-owned foreign keys to users must use `text` references unless the auth schema is changed first.
- Database-backed repositories should accept an explicit schema name and use schema-qualified table names so isolated test schemas can exercise real SQL without mutating the default development schema.
- Shared backend config must work under both Bun and Node because the Better Auth CLI loads `src/auth/` config outside the Bun runtime.
- App-specific user fields belong on Better Auth's `user.additionalFields`; add the matching SQL migration column and expose reads/writes through `/api/v1/users/me` instead of duplicating auth-owned user persistence elsewhere.
- For patient-owned child backend resources, keep list/create routes nested under `/patients/:patientId/...`, keep child mutations on `/<resource>/:resourceId`, and authorize both through `patient_users` joins that return `404` for inaccessible patient or child records.
- Keep frontend auth requests on relative `/api/v1/auth/*` URLs with `credentials: "include"` and let the frontend dev server proxy `/api` to the backend so cookie-backed sessions behave the same in browser verification and local development.
- Run frontend quality checks from `frontend/` with `bun run typecheck`, `bun run lint`, and `bun run test`; use `bun run build` when touching Quasar/Vite config or styling setup.
- Keep public auth routes under `/auth`, protected shell routes under `/app`, and centralize access rules in `frontend/src/router/guards.ts` instead of duplicating checks in pages.
- Keep frontend labels in `frontend/src/i18n/` from the start; Biome currently lints TypeScript and config files while `vue-tsc` covers Vue SFC safety.
- The frontend auth store now owns authenticated profile hydration: restore `/api/v1/auth/session` first, then fetch `/api/v1/users/me` so shell components can rely on app-specific user fields such as `preferredLanguage`.
- Patient-context frontend screens should route through `/app/patients/:patientId/...` and derive shell overview links from the active route params so the same shell can navigate correctly after create and open actions.
- Keep patient-owned child resources in their own frontend domain modules and stores, then compose them inside patient overview pages instead of turning the patients store into a catch-all workspace store.
- Preserve doctor-authored narrative fields such as medical instruction `original_notes` exactly as submitted; trim optional metadata fields separately so filters and labels stay normalized without mutating the clinical note text.
- For patient-owned child resources that need both list filtering and deeper narrative review, keep filtered list state in a domain store, compose the list into the patient overview, and add a dedicated `/app/patients/:patientId/<resource>/:resourceId` detail route for the full record.
- For patient-owned child records with optional references to other patient-owned records, validate those links against the same `patient_id` in the repository layer; plain foreign keys do not stop cross-patient UUID mismatches.
- For tasks, keep stored workflow status separate from derived blocked state: compute incomplete prerequisite IDs in the repository, expose them in API responses, and reject dependency additions that would create cross-patient or circular graphs.
- On the frontend, task editors must treat `/api/v1/tasks/:taskId` and `/api/v1/tasks/:taskId/status` as separate writes, because the backend derives `completedAt` and enforces blocked-task completion rules through the dedicated status endpoint.
- For patient child-resource action rows that mix conditional Quasar `q-btn` components, assign stable `:key` values to each button; otherwise Vue can reuse a sibling button instance after a workflow transition and leave the wrong click handler attached.
- When repositories query custom PostgreSQL enum columns in temporary test schemas, schema-qualify the enum type in casts as well as the table names or list filters will fail outside the configured `search_path`.
- Treat facilities as shared reference data: keep `/api/v1/facilities` as authenticated top-level routes, but keep bookings nested under `/patients/:patientId/bookings` and validate patient-owned task and prescription links in the booking repository.
- For booking UI, load facilities through the bookings store alongside patient bookings and create inline facilities through that same store before saving the booking so the facility select stays synchronized without a second fetch.
- Patient overview summary endpoints should stay in the patients module and aggregate existing patient child tables read-only in the patients repository, so dashboard payloads reuse source records instead of introducing duplicate workflow projections.
- Keep the patient overview snapshot in the patients store and refresh it after patient child-resource mutations; otherwise the top-level care-summary cards drift behind the detail sections.
