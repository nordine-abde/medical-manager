## Codebase Patterns
- In Vue `script setup`, helpers used by top-level computed state or immediate watchers must be declared earlier or written as function declarations; later `const` helpers can throw during initial page setup before they initialize.
- When a Quasar `q-input` uses `clearable`, the bound model can become `null`; page-level filter state should either type that field as `string | null` or normalize with `?? ""` before calling string helpers inside computed state.
- Keep profile reads and writes behind `/api/v1/users/me`; when forwarding updates to Better Auth, map app-facing `fullName` to Better Auth's `name` field and require the session before proxying so unauthenticated updates still return the backend's standard `401` payload.
- When a backend wrapper proxies Better Auth mutations through `authInstance.handler()`, set the forwarded `origin` header to `betterAuthConfig.baseUrl`; the frontend dev server proxy (`http://127.0.0.1:9000`) otherwise trips Better Auth's origin check even though the browser is using the backend wrapper route.
- Keep frontend profile editing behind `src/modules/auth/store.ts` too: settings screens should patch `/api/v1/users/me` through a shared store action and sync form state directly from `authStore.user` on load so persisted profile values repopulate after a full reload.
- Keep patient sharing in the patients module: POST `/api/v1/patients/:patientId/users` accepts a single `identifier` that can be either a user ID or email, duplicate memberships return `409 patient_user_already_linked`, and removing a membership must immediately rely on the same `patient_users` join so future patient reads fall back to `404 patient_not_found`.
- Keep patient sharing UI behind `src/modules/patients/store.ts`: load/add/remove collaborator actions there and disable self-removal in the overview UI so the current caregiver cannot accidentally revoke their own patient access mid-session.
- Vue i18n treats raw `@` characters inside locale strings as linked-message syntax, so email examples in translated placeholders or helper copy should avoid literal `@` characters unless they are escaped intentionally.
- When a patient-owned child relation must reject cross-patient links at the database layer, add a composite foreign key that includes `patient_id` on both sides instead of a bare UUID reference; `prescriptions(patient_id, medication_id) -> medications(patient_id, id)` is the current pattern.
- If a new patient-owned table needs a composite foreign key back to an older patient-owned table, add `unique (patient_id, id)` on the referenced table in the same migration before declaring the new composite foreign key.
- For medication APIs, keep linked prescription context read-only and derive it from `prescriptions.medication_id` in the medications module; the medication row stays the source of renewal fields while prescription state remains in the prescriptions table.
- Keep medication UI in `src/modules/medications/` and let patient overview compose that store with the conditions store; medication forms should edit only medication-owned fields while linked prescriptions render as read-only chips from `medication.linkedPrescriptions`.
- For medication renewals, link tasks through `tasks.medication_id`; renewal tasks should default their due date from `medications.next_gp_contact_date` when no explicit due date is supplied, and `autoRecurrenceEnabled` must stay opt-in.
- Keep patient overview teasers lightweight: aggregate medication renewal highlights through `/api/v1/patients/:patientId/overview`, then use an in-page jump into the full medications workspace instead of duplicating another management surface.
- For document metadata, use a `documents` table with patient ownership plus polymorphic `related_entity_type` and `related_entity_id` fields instead of entity-specific foreign keys; this keeps the schema ready for future `care_event` attachments, so the eventual documents module must validate concrete entity existence and same-patient ownership in application code.
- Keep document binaries in a configurable filesystem root outside backend build output, and generate dated internal storage paths from an allowlisted MIME-to-extension map instead of the uploaded filename.
- For document APIs, validate patient access and same-patient related-entity ownership before storing the binary, keep patient-scoped list and upload routes under `/patients/:patientId/documents`, and expose metadata plus downloads separately as `/documents/:documentId` and `/documents/:documentId/download`.
- Keep patient documents in `src/modules/documents/`; the documents page should compose existing patient child-resource stores to build related-entity labels and selectors, while downloads should use the backend-returned `downloadUrl` directly instead of reconstructing routes in the browser.
- Keep care events in their own backend module with the same patient-child route split as bookings: list/create under `/patients/:patientId/care-events`, get/update under `/care-events/:careEventId`, and validate optional `taskId` and `bookingId` against the same `patient_id` in the repository before writing.
- When a patient-context page links back into async sections on `PatientOverviewPage.vue`, keep the URL hash for shareable deep links but re-run `scrollIntoView()` after the overview data finishes loading; router-level hash scrolling alone can fire before task or booking cards mount and silently miss the target.
- Keep the global caregiver task workspace in the tasks module as a top-level `/tasks` query with high-level `state` filters and embedded patient summary data, rather than adding another patients aggregate route for cross-patient task lists.
- Keep patient timeline aggregation in the patients module as `/patients/:patientId/timeline`; build it as a union over existing patient-owned tables and normalize each entry to a shared `relatedEntity { type, id }` shape so timeline UIs can navigate without resource-specific parsing.
- Keep the global `/app/timeline` workspace in the patients frontend module too: load `/api/v1/timeline` through `src/modules/patients/store.ts`, then reuse the same related-entity to route or hash mapping as patient timeline pages instead of inventing cross-patient detail routes.
- For notification persistence, keep patient reminder preferences in `notification_rules` with a unique `(patient_id, rule_type)` key, and store outbound delivery attempts in `notification_logs` with nullable workflow links plus a status-driven `sent_at` check so pending rows stay unsent while sent and failed rows always carry their delivery timestamp.
- Keep reminder generation in the backend notifications module as a top-level authenticated job-style route; derive candidates directly from tasks, bookings, and medications, and make the run idempotent by skipping any trigger that already has a pending, sent, or failed notification log.
- Keep Telegram delivery in the notifications module by reusing pending `notification_logs` as the send queue; persist the Telegram response message id on `notification_logs.external_message_id` and record failures on the same row instead of introducing a second delivery ledger.
- Keep scheduled reminder or recurrence runs in the notifications module as authenticated job-style routes; recurring task generation should clone completed source tasks only when `auto_recurrence_enabled = true`, preserve recurrence fields on the new task, and stay idempotent by checking for an existing same-window task with the same patient-scoped identity plus next due date.
- Keep patient notification settings as a patient-context frontend module backed by `/api/v1/patients/:patientId/notifications/settings`; the UI should edit one shared Telegram chat ID plus per-rule toggles or lead times, while the backend fans that payload out to the three `notification_rules` rows.
- When localizing established shell pages, do not stop at headings and buttons; also move summary-row labels, helper hints, disabled-state guidance, and small metadata prefixes into `src/i18n/*` so browser verification does not leave mixed-language fragments behind.
- Keep patient notification delivery actions inside `src/modules/notifications/store.ts`; pages should trigger `/api/v1/notifications/deliveries/process` through the store and derive patient-specific feedback from the returned log list instead of making one-off page fetches.
